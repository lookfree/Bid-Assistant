三、核心概念
- 账户(user)：手机号注册登录。
- 套餐(plan)：一条配置记录，定义"价格 / 计费周期 / 每周期赠送积分 / 权益开关 / 并发或项目数上限"等字段。档位数量不固定，由配置增减。
- 订阅(subscription)：用户当前所处套餐、周期、状态（无自动续费；到期靠提醒+手动续费）。
- 积分账本(credit ledger)：append-only 事件流水，余额 = 流水之和。
- 订单(payment_order) / 退款(refund)。
- 支付终端(payment_terminal)：收钱吧交易终端凭证（terminal_sn/terminal_key），业务接口签名身份，需持久化并每日签到轮换密钥。
- 推荐关系(referral)：邀请人↔被邀请人，及奖励发放记录。

四、数据模型（字段给全，数值留空由配置注入）
1. plans：id、name、price、currency、billing_cycle、grant_credits_per_cycle、features(JSON 权益开关)、limits(JSON)、status(上架/下架)、version。★所有数值字段由运营后台/配置填写。
2. subscriptions：user_id、plan_id、status(active/past_due/expired)、current_period_start/end。（无 auto_renew/agreement_no——不做自动续费）
3. credit_transactions（只追加）：id、user_id、type(grant/purchase/hold/settle/release/expire/referral_reward/refund_clawback——退款注销已入账积分，负向)、amount(±)、source_batch、expire_at、ref(关联 order/agent_run/referral)、idempotency_key、created_at。
4. credit_balances：user_id、balance（余额缓存，用于对账，权威仍是流水之和）。
5. payment_orders：id、user_id、type(recharge/purchase/renewal)、amount、status(created/paid/failed/unknown/refunded)、provider(固定 shouqianba)、client_sn(我方订单号，全局唯一，送收钱吧)、provider_trade_no(收钱吧订单号 sn)、channel_trade_no(微信/支付宝渠道单号 trade_no)、payway(实际付款方式)、idempotency_key、created_at。
6. payment_terminals：id、terminal_sn、terminal_key（加密存储）、device_id、activated_at、last_checkin_at。★激活/每日签到产生的终端参数唯一真相，多实例集群共享；密钥丢失只能重激活。
7. refunds：id、order_id、amount、reason、status、operator、created_at。（走收钱吧退款接口，支持部分退款）
8. referrals：inviter_id、invitee_id、code、status、reward_state(pending/unlocked/capped)、created_at。
9. 配置表 billing_configs：key-value 形式存放"积分消耗口径、汇率、充值包、推荐奖励规则、过期天数、到期提醒天数档、支付轮询窗口"等所有可调参数（见第十节）。

五、积分账本机制（核心）
- 余额 = Σ流水，不维护可随手改的余额数字（balance 仅作缓存与对账）。
- 每次 AI 操作走"预扣→结算"两段式：
  1. 校验余额充足 → 写 hold(-N) 预扣（冻结，防并发超扣）；N 由配置的"操作积分口径"决定。
  2. 操作成功 → 按实际用量写 settle，多退少补；失败 → 写 release(+N) 全额退还。
- 所有扣减/回调必须带 idempotency_key，保证重复请求不重复扣。
- 多来源积分按"先过期先扣"消耗（FIFO by expire_at）。
- 过期：定时任务扫描到期的赠送/奖励积分，写 expire 流水。

六、单笔支付（充值 / 购买 / 续费，收钱吧 C 扫 B 跳转支付）
0. 前置：终端已激活（vendor 参数 + 门店激活码 + device_id → terminal_sn/terminal_key，落 payment_terminals）；每日签到轮换 terminal_key 并更新落库。
1. 用户选充值包或套餐 → 创建 payment_orders(created, 带幂等键, client_sn 全局唯一)，金额取自配置（单位：分）。
2. 按收钱吧 WAP2 规则拼跳转支付 URL（client_sn/total_amount/subject/return_url/notify_url + 参数 ASCII 升序 & 拼接 + "&key="+terminal_key 的 MD5 大写签名）→ 转二维码展示。
3. 用户用微信/支付宝扫码 → 收钱吧 H5 确认页付款 → 跳回 return_url。
4. 结果获取双通道（先到终态者生效，幂等合一）：
   a. 回调 notify_url：用收钱吧公钥做 SHA256WithRSA 验签（Authorization 头带签名，body 为原文）→ 订单置 paid → 写积分 purchase 流水 / 激活或续期会员。
   b. 轮询兜底：订单有效期 4 分钟；后端调收钱吧「查询」接口轮询（0–1 分钟每 3s、1–5 分钟每 10s、第 6 分钟最后一次），仍无终态 → 订单置 unknown 待对账。
5. 幂等关键：回调可能重复、且与轮询并发，按收钱吧 sn + 订单状态机保证只入账一次；trade_no/payway 一并落库供对账。
6. 监管合规：请求头 X-Forwarded-For 送顾客手机真实公网 IP；生产网关即真实交易（测试激活码对应测试商户号）。

七、会员到期与续费（无自动续费）
- 到期提醒：定时任务（锁内执行）扫描 current_period_end 临近（T-7/T-3/T-1，天数档可配）且 status=active 的订阅 → 推送提醒（短信/站内信）；同一订阅同一档只提醒一次（幂等键=订阅+提醒档）。
- 手动续费：用户在会员中心点「续费」→ 走第六节单笔支付（type=renewal）→ 支付成功 → 续期 current_period_end、发放当期赠送积分(grant)。
- 到期未续费：status: active → past_due（宽限期，天数可配）→ expired；权益降级为免费版；积分按流水规则保留/过期。
- 明确不做：签约/代扣/周期扣款/自动续费及其重试状态机。

八、推荐奖励机制（通用，规则全配置化）
实现一套通用的邀请奖励引擎，具体奖励数值/解锁条件/封顶/有效期全部来自配置：
- 邀请人/被邀请人各自的奖励额度 = 配置值。
- 支持"立即发放"与"延迟解锁"两段（解锁触发条件如"被邀请人首次付费"由配置开关决定）。
- 单用户奖励封顶 = 配置值，达上限后 reward_state=capped。
- 奖励积分写 referral_reward 流水，带有效期（配置）。
- 防刷：手机号/实名/设备唯一校验；异常邀请（同 IP 段、集中时段、注册即弃）标记冻结，留风控审计。

九、对账、退款、风控、审计
- 每日对账定时任务：按 payment_orders 逐笔调收钱吧「查询」接口核对终态（重点清算 unknown 态订单）vs credit_transactions，差异告警；如开通收钱吧对账单导出则以账单文件核对替代逐笔查询。
- 积分账本可独立审计：余额 = Σ流水。
- 退款：走 refunds + 调收钱吧退款接口（支持部分退款）+ 扣减积分/降级权益，按平台规则。
- 终端密钥安全：terminal_key 加密落库；签到失败按排障流程处理（网络/程序排查 → 重签到 → 极端情况重激活）；vendor_key/收钱吧公钥只存 env/密管，不入 git。
- 所有敏感操作（改套餐/调积分/退款/手动发放奖励）留操作审计日志（操作人、时间、前后值）。
